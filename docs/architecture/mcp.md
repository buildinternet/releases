# Remote MCP Server

The MCP Worker (`workers/mcp/`) exposes a remote MCP server at `mcp.releases.sh` using Cloudflare's `createMcpHandler` with Streamable HTTP transport. It provides read-only tools across three surfaces — search (`search`, `get_latest_releases`, `get_release`), registry detail (`list_catalog`, `get_catalog_entry`, `list_organizations`, `get_organization`, `lookup_domain`), and AI analysis (`summarize_changes`, `compare_products`, gated behind `ENABLE_AI_TOOLS=true`). Alongside the tool surface it advertises MCP **resources** (`releases://org/...`, `releases://catalog/...`) for browseable access and **prompts** (`whats_new`, `compare_products`, `catch_me_up`) as priming entry points. Read tools are public — no token required. Privileged paths (the AI tools and the on-demand lookup) require a `write`-scoped API token; see [Authentication & scope enforcement](#authentication--scope-enforcement).

**Catalog unification (issue #539):** `list_catalog` and `get_catalog_entry` treat products and standalone sources as one thing — a catalog entry — and fold them into a single list with an `entryType: "product" | "source"` discriminator. `search` is the unified entry point: it searches orgs, catalog, and releases in one call, and takes an optional `type: ("orgs" | "catalog" | "releases")[]` filter to skip sections (agents that only need registry lookups should pass `type: ["catalog"]` to avoid the vector-retrieval path). The per-surface tools (`search_releases`, `search_registry`, `list_sources`, `list_products`, `get_product`) remain as deprecated shims for one release cycle — their titles and descriptions mark them `(deprecated)`. The `releases://catalog/{slug}` resource mirrors the tool shape; `releases://product/...` and `releases://source/...` stay for one cycle as deprecated aliases.

**Progressive detail (issue #556):** `get_organization` and `get_catalog_entry` follow a simple/expanded pattern. By default both return the lightweight view — `get_organization` shows a preview of the AI-generated overview (first paragraph), and `get_catalog_entry` for a source lists tracked CHANGELOG files by path and byte size. Pass `include_overview: true` on `get_organization` to inline the full briefing; pass `include_changelog: true` (or any of `changelog_path` / `changelog_offset` / `changelog_limit` / `changelog_tokens`) on `get_catalog_entry` to embed the CHANGELOG slice. Heading-aligned slicing supports per-package files (e.g. `packages/next/CHANGELOG.md`) and emits `totalTokens` / `sliceTokens` for LLM budgeting (recommended brackets: 2000 / 5000 / 10000 / 20000). This replaces the standalone `get_organization_overview`, `get_source`, and `get_source_changelog` tools.

The worker binds to the same D1 database as the API and discovery workers and carries a service binding (`API`) to the API worker so it can forward on-demand lookup calls without a round-trip through the public internet (see `"services": [{ "binding": "API", ... }]` in `workers/mcp/wrangler.jsonc`). AI tools (`summarize_changes`, `compare_products`) use an `ANTHROPIC_API_KEY` from the secrets store. Like the discovery worker, `workers/mcp/` is excluded from root `workspaces` to avoid import conflicts.

**On-demand lookup fallback (issues #611, #662, #664):** When `search` returns no entity hits (orgs / catalog) and the query parses as a `{org}/{repo}` GitHub coordinate — optionally prefixed `github:` — the tool calls `POST /v1/lookups` via the `API` service binding before returning. The gate intentionally ignores release / chunk hits: those routinely fire on a single segment token (e.g. "shopify" matching another org's release body) and would otherwise suppress the lookup for the typed repo. `search_releases` has no entity buckets and always attempts the lookup on coordinate-shaped input. The lookup result is merged under a `lookup` field in the tool response (`status`, `source`, `releases`, `relatedOrg`). Five outcomes: `indexed` (new source materialized), `existing` (already tracked — matched case-insensitively against `sources.url`, so `shopify/toxiproxy` and `Shopify/Toxiproxy` resolve to the same row), `empty` (repo exists but has no releases/CHANGELOG), `not_found` (404/403/archived), `deferred` (GitHub rate-limit or 5xx — transient, no negative cache written). The `lookup` field is `null` when no coordinate was detected or when an entity match was already returned. The on-demand lookup is also suppressed when `search` is called with a `domain` filter — the caller has already named an entity.

**Domain lookup (`lookup_domain`):** Pure resolution — given a URL-shaped input, normalize it via `normalizeDomain` (`@buildinternet/releases-core/domain`) and exact-match against `organizations.domain` (primary) and `domain_aliases.domain` (alias for either an org or a product). Returns the matched org plus any products whose alias targets the same domain. Distinct from the GitHub coordinate path: there is no probing or materialization for domains — unknown domains surface a "no match" message. The same normalizer powers the `domain` input on `search`, which scopes a query to the resolved org (and short-circuits with empty arrays + a "no match" hint when the domain isn't owned).

Deploy: `bun run deploy:mcp`. Dev: `bun run dev:mcp`. Connect from Claude Desktop: `npx mcp-remote https://mcp.releases.sh/mcp`.

## Authentication & scope enforcement

Scoped API tokens (Phase 2 of the [scoped API tokens design](../superpowers/specs/2026-05-20-scoped-api-tokens-design.md)) are enforced here. `workers/mcp/src/auth.ts` resolves the caller's identity once per request, **at the HTTP boundary** in `index.ts` — `createMcpHandler` doesn't expose the inbound `Authorization` header to tool handlers, so the boundary is the only place to read it. Identity (scopes + the raw token) is threaded into `createServer` via `CreateServerOptions`, the same mechanism that carries `userAgent`.

**Identity resolution** (`resolveMcpAuth`, mirroring the API worker's `resolveAuth`):

- A `relk_…` Bearer is verified against D1 via the shared `verifyApiToken` from `@releases/core-internal/api-token-store` — one verification path for both workers. On success the caller carries the token's scopes; an invalid/unknown `relk_` token is **ignored** (resolves to anonymous read) so public reads never 401, exactly like the API worker's public-read path. Gated by `API_TOKENS_DISABLED`.
- A `relu_…` Bearer (Better Auth user key) is verified by authenticating it against the API worker's `GET /v1/tokens/me` over the `API` service binding — the MCP worker never instantiates Better Auth. The API worker's middleware verifies **and meters** the key exactly once and returns its scopes. Metering is gated on a JSON-RPC method peek (`isMeteredMcpMethod`): only billable methods (`tools/call`, and anything not in the protocol-overhead allowlist) trigger the verify; `initialize`/`tools/list`/`ping`/`notifications/*`/etc. resolve to anonymous and are never metered. A `429` from `/v1/tokens/me` surfaces as a `429` from the boundary; an invalid key, an unreachable binding, the flag being off, or any error fail-open to anonymous read. Gated by `user-api-keys-enabled` (and `API_TOKENS_DISABLED`); inert in prod until Phase 3.
- A JWT-shaped Bearer is treated as a "Sign in with Releases" OAuth access token (#1483) and verified locally with `jose` + the AS JWKS via the shared `@releases/lib/oauth-jwt` (the MCP worker never imports better-auth — zod-pin). A verified token carries the `scope`-claim scopes (already role-clamped at issuance) under a `oauth_<sub>` tokenId with `token: null`. Unlike the `relk_`/`relu_` lanes, a presented-but-**invalid** JWT (bad signature, wrong `iss`/`aud`, expired, or zero API scopes) does **not** fall open to anonymous — it returns a `401` + `WWW-Authenticate` discovery challenge (see [OAuth discovery surface](#oauth-discovery-surface-rfc-9728)). Issuer/audience come from `OAUTH_JWT_ISSUER`/`OAUTH_JWT_AUDIENCE` (prod defaults shared with `well-known.ts`).
- The static `RELEASES_API_KEY` presented as Bearer maps to **root** (`["*"]`).
- Anything else (**no credential**) is **anonymous**, carrying an implicit `["read"]` scope — public read is never gated.

A successful `relk_` token use records `last_used_at` via `touchLastUsed` (throttled to 60s, fire-and-forget through `waitUntil`) so the admin surface audits usage across both workers. `relu_` user keys are metered by Better Auth's `apikey` table instead, so the machine-lane write is skipped for them (`machineTokenIdForUsage`).

**Per-tool scope map** (`requireScope` wrapper in `mcp-agent.ts`):

| Tools                                                                | Required scope                  |
| -------------------------------------------------------------------- | ------------------------------- |
| All read tools (`search`, `get_*`, `list_*`, `lookup_domain`)        | none (open to anonymous `read`) |
| `summarize_changes`, `compare_products` (gated by `ENABLE_AI_TOOLS`) | `write`                         |
| The on-demand `/v1/lookups` fallback inside `search`                 | `write`                         |

Under-scoped AI-tool calls return a tool result with `isError: true` and an `insufficient_scope` message (a tool-level error the model can read and adapt to, not a protocol abort). Note: user (`relu_`) keys are capped at read (`USER_API_KEY_MAX_SCOPE`), so they can never satisfy `write` and are therefore permanently excluded from the AI tools and the on-demand lookup — only `relk_` write/admin tokens and root reach them.

**Confused-deputy fix:** the on-demand lookup (`maybeLookup`) materializes a hidden source — a write. It previously presented the static root key on every call, so any MCP client borrowed root downstream. It now fires only when the caller's scopes satisfy `write`, and forwards the **caller's own** `relk_` token to `POST /v1/lookups` (a static-root caller forwards root — root acting as root). Anonymous/read callers no longer trigger the lookup at all. **Behavior change:** on-demand GitHub indexing via the public MCP now requires a `write`-scoped token rather than silently running as root for everyone. For a `relu_` caller the identity carries `token: null`, so `maybeLookup` forwards the **root** key (its `authToken ?? rootKey` fallback) rather than the user key — the inbound tool call is already metered once at the boundary, and forwarding the user key would double-count it at `/v1/lookups`. Note: under the current read-only ceiling (`USER_API_KEY_MAX_SCOPE = "read"`) a `relu_` key's scopes are clamped to read at auth time, so it can never satisfy this `write` gate — `maybeLookup` never fires for a user key at all, and the forwarding-root path above only becomes reachable if that ceiling is later raised.

**Staging gate bridge:** the staging access gate (`STAGING_ACCESS_KEY`, staging only) runs in the same `resolveMcpAuth` pass. It still accepts the `X-Releases-Staging-Key` header and `Authorization: Bearer <staging-key>`, and now **also** accepts a valid staging-DB `relk_` token (or the root key) — so an Anthropic managed agent, whose vault credentials are Bearer-only, can authenticate to `mcp-staging` with a token instead of the shared staging key. A `relu_` user identity does **not** open the gate (it carries `token: null`; the bridge requires `identity.token !== null`), so a user key must still supply the staging key in staging. CORS preflight (`OPTIONS`) always passes.

## OAuth discovery surface (RFC 9728)

The MCP authorization spec (2025-06-18) requires an MCP server, acting as an OAuth 2.1 resource server, to advertise its authorization server so a client can discover where to authenticate and what `resource` (RFC 8707) to request. The better-auth AS only mints a JWT (rather than an opaque token) when the client sends `resource` on the token request — so without this discovery surface the JWT lane above never engages for an off-the-shelf client. `workers/mcp/src/well-known.ts` provides it:

- **`GET /.well-known/oauth-protected-resource`** (and the RFC 9728 §3.1 path-suffixed `…/oauth-protected-resource/mcp`) returns the protected-resource metadata: `{ resource, authorization_servers, scopes_supported, bearer_methods_supported }`. `resource` is the configured audience (`OAUTH_JWT_AUDIENCE`, the bare origin `https://mcp.releases.sh`) so the value a compliant client echoes back as `resource` equals the `aud` this resource server verifies; `authorization_servers` is `[OAUTH_JWT_ISSUER]` (`https://api.releases.sh/api/auth`), from which the client does RFC 8414 AS-metadata discovery. In prod it is public discovery, short-circuited **before** `resolveMcpAuth` (like `/robots.txt`); on staging it is deferred until **after** the staging access gate so it stays opaque like every other route (a harness already carries the staging key).
- **`WWW-Authenticate` challenge:** a presented-but-invalid OAuth JWT returns `401` with `WWW-Authenticate: Bearer error="invalid_token", resource_metadata="<origin>/.well-known/oauth-protected-resource"`, pointing the client at the metadata to re-authenticate. On staging the access gate runs first (`resolveMcpAuth`), so an invalid credential without the staging key gets the generic staging `401` rather than the discovery hint.

**Deliberate stance — anonymous read is preserved.** The MCP server is a public changelog registry, so this is _not_ a fully-protected resource: a **tokenless** request still resolves to anonymous `read` and never receives a 401. Only a request that presents an invalid OAuth credential is challenged. This intentionally diverges from the spec's "401 on any tokenless connect" model, which would break public read.

The REST API worker is also an OAuth resource server and serves its own RFC 9728 metadata at `api.releases.sh/.well-known/oauth-protected-resource` (`resource` = the API origin); see [remote-mode.md → Auth model](remote-mode.md#auth-model). Its server-to-server JWKS fetch is why the staging access gate exempts `/api/auth/jwks`.

## MCP App UIs (`workers/mcp/ui/`)

Hosts that support the MCP Apps extension (Claude Desktop is the first) render a tool's response as an interactive UI when the tool advertises `_meta.ui.resourceUri`. Each UI is a single self-contained HTML resource registered alongside the existing entity resources. Hosts without UI support fall back to the markdown text in `content[0].text`, so every UI-enabled tool keeps its text rendering intact.

**Live surfaces:**

- `get_latest_releases`, `get_collection_releases` → `ui://releases/release-feed.html` (card timeline with cursor-based "Load more", "Open source", and "Ask about this").

**Build pattern (Bun, no Vite):** Each app lives in its own folder under `workers/mcp/ui/` with an `app.tsx` entry and a `styles.css`. `workers/mcp/ui/build.ts` runs `Bun.build` over the entry, reads the sibling CSS, inlines both into a minimal HTML shell, and writes the bundled strings into `workers/mcp/src/ui-bundles.ts`. The worker imports those strings and serves them through `registerAppResource` in `workers/mcp/src/resources.ts`. The bundle file is committed so deploys never depend on a UI install; `deploy:mcp` regenerates it first as a safety net.

**Adding a new UI:**

1. `mkdir workers/mcp/ui/<name>` and drop in `app.tsx` (React entry) + `styles.css`.
2. Add an entry to the `APPS` array in `workers/mcp/ui/build.ts`.
3. Add an exported URI constant + `registerAppResource` call in `workers/mcp/src/resources.ts`.
4. On the tool registration, add `_meta: { ui: { resourceUri: <CONST> } }`.
5. Have the tool's handler also return `structuredContent` with the typed payload the UI reads (see `ReleaseFeedStructured` in `workers/mcp/src/tools.ts` for the shape).
6. `bun run mcp:ui:build` to regenerate the bundle.

**Aliases:**

- `bun run mcp:ui:build` — production bundle, commits-clean
- `bun run mcp:ui:build:dev` — inline source maps + unminified for debugging
- `bun run mcp:ui:watch` — rebuild on save during UI iteration
- `bun run mcp:ui:typecheck` — `tsc --noEmit` over the UI tree

**Data contract:** UI-enabled tools attach `structuredContent` to their result. The UI consumes that field directly via `app.ontoolresult`; the model still reads the markdown `content[0].text`. Cursor pagination from the UI is wired through `app.callServerTool({ name, arguments: { ...inputs, cursor } })` — no host re-prompt needed to extend the feed.

**Vite note:** the MCP Apps SDK's React template ships with Vite + `vite-plugin-singlefile`. We don't use Vite — Bun's bundler produces the same single-file output via a ~30-line build script. The `@modelcontextprotocol/ext-apps` package is consumed only by UI code (for `useApp` and types); the worker doesn't depend on it at runtime, so worker deploys stay lean.

## Spec utilities

- **Ping (`ping`):** supported transparently. The SDK's base `Protocol` class registers a default handler that responds with `{}`, so every server instance ships it without code.
- **Tool annotations:** every tool carries `readOnlyHint: true, destructiveHint: false, openWorldHint: false` plus a display `title`. Data-read tools are `idempotentHint: true`; the AI tools (`summarize_changes`, `compare_products`) flip it off because LLM output varies per call. The hints are asserted in `tests/unit/mcp-tool-annotations.test.ts` so regressions fail fast.
- **Identifiers — IDs alongside slugs:** every tool, prompt, and resource that takes an org / product / source identifier accepts a typed ID (`org_…`, `prod_…`, `src_…`) interchangeably with the slug or `org/slug` coordinate form. Releases are different — `get_release` accepts only `rel_…` or a bare 21-char nanoid, never a slug. Tool descriptions explicitly list each shape. The bare-slug path on `releases://catalog/{slug}` and `releases://org/{orgSlug}` still works, but the URI segment also resolves a typed ID — completion only offers slugs because IDs aren't browseable. `findOrg` (`workers/mcp/src/tools.ts`) folds the `org_…` PK match into the existing UNION query as an additional `OR` predicate, so the indexed PK hit is a fast-path within the same statement that handles slug / domain / domain-alias / account-handle resolution. The catalog and product helpers branch on `getEntityType` to dispatch to the correct lookup. Prefer IDs in agent prompts where stability matters more than readability.
- **Resources + completion:** `workers/mcp/src/resources.ts` registers four templates — `releases://org/{orgSlug}`, `releases://catalog/{slug}`, and the deprecated-alias pair `releases://product/{productSlug}` + `releases://source/{sourceSlug}`. All four are completion-only (no `resources/list` enumeration) — the SDK doesn't thread a cursor through `listCallback`, so capping or alphabetizing would silently hide rows as the catalog grows. Completion (in `workers/mcp/src/slug-completion.ts`) runs case-insensitive substring LIKE against both the slug and the display name, ranks prefix matches ahead of mid-string matches, sanitizes user input (`%`/`_`), and returns up to 20 candidates. **Org completion returns bare slugs** because `organizations.slug` is still globally unique. **Catalog (product + source) completion returns `org/slug` coordinates** instead of bare slugs because per-org slug uniqueness (#690) means a bare slug isn't an unambiguous handle, and downstream tools (`summarize_changes`, `compare_products`, `get_latest_releases`, `get_catalog_entry`) reject bare slugs with a 400 — coordinates round-trip cleanly. Typed IDs are not surfaced through completion — they're not memorable, but URI handlers and tool inputs accept them directly. `completeCatalogSlug` fans out to the product and source completers in parallel and de-dupes on collision. Substring LIKE forces a table scan but the tables are low-thousands of rows — revisit with FTS5 if the catalog grows by 100×. Read callbacks delegate to `getOrganization`/`getCatalogEntry`/`getProduct` and return `text/markdown`; the deprecated source-resource route is folded into `getCatalogEntry` under the hood.
- **Prompts:** `workers/mcp/src/prompts.ts` registers three priming prompts — `whats_new`, `compare_products`, `catch_me_up`. `whats_new` and `compare_products` accept a `prod_…` id or `org/slug` coordinate (completion returns coordinates so the interpolated tool call is accepted); `catch_me_up` accepts an org\_ id, slug, domain, name, or account handle. Prompts emit a single user message that names the exact tools the LLM should call (so models that don't auto-plan still land on the right entry points). `whats_new` and `compare_products` prefer `summarize_changes`/`compare_products` when `ENABLE_AI_TOOLS=true` and fall back to `get_latest_releases` otherwise. Completion for both resources and prompts reuses `workers/mcp/src/slug-completion.ts`.

## Parity with WebMCP

The web app also exposes a browser-side subset of these tools via `navigator.modelContext.registerTool()` — see `web/src/components/webmcp-provider.tsx`. Today that surface mirrors `search`, `lookup_domain`, `list_organizations`, `get_organization`, `get_catalog_entry`, `get_release` plus an `open_search_page` nav helper, all pointed at the public API. WebMCP is intentionally a lightweight subset — it does not mirror resources, prompts, or the AI tools, and it is not expected to track remote-MCP surface additions 1:1. When you add, rename, or change the signature of a read-only tool in `workers/mcp/src/tools.ts`, update the WebMCP provider in the same PR so the tool subset doesn't drift. The OSS CLI's stdio bridge (`releases admin mcp serve`) is a thin proxy to the hosted server, so it inherits tool changes on the next worker deploy. Write/admin tools stay out of WebMCP — browser-side callers can't present an API key safely.

## MCP Registry listing

The server is registered as `sh.releases/mcp` in the official MCP Registry via HTTP-domain auth against `releases.sh` (proof file lives at `web/public/.well-known/mcp-registry-auth`). Metadata is in `workers/mcp/server.json` — bump `version` when you want to publish an update. The `Deploy Workers` GitHub Action runs `mcp-publisher publish` automatically on merges that touch `server.json` (gated on the `MCP_REGISTRY_PRIVATE_KEY_PEM` repo secret). Manual publish: `bun run publish:mcp-registry` after `mcp-publisher login http --domain releases.sh --private-key <hex>`.
