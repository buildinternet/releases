# Remote MCP Server

The MCP Worker (`workers/mcp/`) exposes a remote MCP server at `mcp.releases.sh` using Cloudflare's `createMcpHandler` with Streamable HTTP transport. It provides read-only tools across three surfaces — search (`search`, `get_latest_releases`, `get_release`), registry detail (`list_catalog`, `get_catalog_entry`, `list_organizations`, `get_organization`), and AI analysis (`summarize_changes`, `compare_products`, gated behind `ENABLE_AI_TOOLS=true`). Alongside the tool surface it advertises MCP **resources** (`releases://org/...`, `releases://catalog/...`) for browseable access and **prompts** (`whats_new`, `compare_products`, `catch_me_up`) as priming entry points. No authentication required — all read surfaces are public.

**Catalog unification (issue #539):** `list_catalog` and `get_catalog_entry` treat products and standalone sources as one thing — a catalog entry — and fold them into a single list with a `kind: "product" | "source"` discriminator. `search` is the unified entry point: it searches orgs, catalog, and releases in one call, and takes an optional `type: ("orgs" | "catalog" | "releases")[]` filter to skip sections (agents that only need registry lookups should pass `type: ["catalog"]` to avoid the vector-retrieval path). The per-surface tools (`search_releases`, `search_registry`, `list_sources`, `list_products`, `get_product`) remain as deprecated shims for one release cycle — their titles and descriptions mark them `(deprecated)`. The `releases://catalog/{slug}` resource mirrors the tool shape; `releases://product/...` and `releases://source/...` stay for one cycle as deprecated aliases.

**Progressive detail (issue #556):** `get_organization` and `get_catalog_entry` follow a simple/expanded pattern. By default both return the lightweight view — `get_organization` shows a preview of the AI-generated overview (first paragraph), and `get_catalog_entry` for a source lists tracked CHANGELOG files by path and byte size. Pass `include_overview: true` on `get_organization` to inline the full briefing; pass `include_changelog: true` (or any of `changelog_path` / `changelog_offset` / `changelog_limit` / `changelog_tokens`) on `get_catalog_entry` to embed the CHANGELOG slice. Heading-aligned slicing supports per-package files (e.g. `packages/next/CHANGELOG.md`) and emits `totalTokens` / `sliceTokens` for LLM budgeting (recommended brackets: 2000 / 5000 / 10000 / 20000). This replaces the standalone `get_organization_overview`, `get_source`, and `get_source_changelog` tools.

The worker binds to the same D1 database as the API and discovery workers and carries a service binding (`API`) to the API worker so it can forward on-demand lookup calls without a round-trip through the public internet (see `"services": [{ "binding": "API", ... }]` in `workers/mcp/wrangler.jsonc`). AI tools (`summarize_changes`, `compare_products`) use an `ANTHROPIC_API_KEY` from the secrets store. Like the discovery worker, `workers/mcp/` is excluded from root `workspaces` to avoid import conflicts.

**On-demand lookup fallback (issue #611):** When `search` or `search_releases` returns no hits and the query parses as a `{org}/{repo}` GitHub coordinate, the tool calls `POST /v1/lookups` via the `API` service binding before returning. The lookup result is merged under a `lookup` field in the tool response (`status`, `source`, `releases`, `relatedOrg`). Five outcomes: `indexed` (new source materialized), `existing` (already tracked), `empty` (repo exists but has no releases/CHANGELOG), `not_found` (404/403/archived), `deferred` (GitHub rate-limit or 5xx — transient, no negative cache written). The `lookup` field is `null` when no coordinate was detected or when the query already returned hits.

Deploy: `bun run deploy:mcp`. Dev: `bun run dev:mcp`. Connect from Claude Desktop: `npx mcp-remote https://mcp.releases.sh/mcp`.

## Spec utilities

- **Ping (`ping`):** supported transparently. The SDK's base `Protocol` class registers a default handler that responds with `{}`, so every server instance ships it without code.
- **Tool annotations:** every tool carries `readOnlyHint: true, destructiveHint: false, openWorldHint: false` plus a display `title`. Data-read tools are `idempotentHint: true`; the AI tools (`summarize_changes`, `compare_products`) flip it off because LLM output varies per call. The hints are asserted in `tests/unit/mcp-tool-annotations.test.ts` so regressions fail fast.
- **Resources + completion:** `workers/mcp/src/resources.ts` registers four templates — `releases://org/{orgSlug}`, `releases://catalog/{slug}`, and the deprecated-alias pair `releases://product/{productSlug}` + `releases://source/{sourceSlug}`. All four are completion-only (no `resources/list` enumeration) — the SDK doesn't thread a cursor through `listCallback`, so capping or alphabetizing would silently hide rows as the catalog grows. Completion (in `workers/mcp/src/slug-completion.ts`) runs case-insensitive substring LIKE against both the slug and the display name, ranks prefix matches ahead of mid-string matches, sanitizes user input (`%`/`_`), and returns up to 20 slugs. `completeCatalogSlug` fans out to the product and source completers in parallel and de-dupes on collision. Substring LIKE forces a table scan but the tables are low-thousands of rows — revisit with FTS5 if the catalog grows by 100×. Read callbacks delegate to `getOrganization`/`getCatalogEntry`/`getProduct` and return `text/markdown`; the deprecated source-resource route is folded into `getCatalogEntry` under the hood.
- **Prompts:** `workers/mcp/src/prompts.ts` registers three priming prompts — `whats_new`, `compare_products`, `catch_me_up` — each with `completable()` slug arguments. Prompts emit a single user message that names the exact tools the LLM should call (so models that don't auto-plan still land on the right entry points). `whats_new` and `compare_products` prefer `summarize_changes`/`compare_products` when `ENABLE_AI_TOOLS=true` and fall back to `get_latest_releases` otherwise. Slug completion for both resources and prompts reuses `workers/mcp/src/slug-completion.ts`.

## Parity with WebMCP

The web app also exposes a browser-side subset of these tools via `navigator.modelContext.registerTool()` — see `web/src/components/webmcp-provider.tsx`. Today that surface mirrors `search`, `list_organizations`, `get_organization`, `get_catalog_entry`, `get_release` plus an `open_search_page` nav helper, all pointed at the public API. WebMCP is intentionally a lightweight subset — it does not mirror resources, prompts, or the AI tools, and it is not expected to track remote-MCP surface additions 1:1. When you add, rename, or change the signature of a read-only tool in `workers/mcp/src/tools.ts`, update the WebMCP provider in the same PR so the tool subset doesn't drift. The OSS CLI's stdio bridge (`releases admin mcp serve`) is a thin proxy to the hosted server, so it inherits tool changes on the next worker deploy. Write/admin tools stay out of WebMCP — browser-side callers can't present an API key safely.

## MCP Registry listing

The server is registered as `sh.releases/mcp` in the official MCP Registry via HTTP-domain auth against `releases.sh` (proof file lives at `web/public/.well-known/mcp-registry-auth`). Metadata is in `workers/mcp/server.json` — bump `version` when you want to publish an update. The `Deploy Workers` GitHub Action runs `mcp-publisher publish` automatically on merges that touch `server.json` (gated on the `MCP_REGISTRY_PRIVATE_KEY_PEM` repo secret). Manual publish: `bun run publish:mcp-registry` after `mcp-publisher login http --domain releases.sh --private-key <hex>`.
