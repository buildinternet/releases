# Bound the Firecrawl-auto backfill path + document the deep-history path

- **Issue:** https://github.com/buildinternet/releases/issues/1271
- **Date:** 2026-05-30
- **Status:** Design — approved
- **Predecessor:** `docs/superpowers/specs/2026-05-30-backfill-source-primitive-design.md` (#1265, PR #1269)

## Problem

`POST /v1/workflows/backfill-source` (shipped in #1269) runs the full backfill **synchronously inside one HTTP request**. A prod dry-run smoke against the live OpenAI Firecrawl source showed an unbounded full-history run does not return within a normal client timeout:

| Run                       | Result                         | Time     |
| ------------------------- | ------------------------------ | -------- |
| Firecrawl, unbounded      | client timed out, 0 bytes      | >180s    |
| Firecrawl, `maxWindows:1` | 200 — 55 extracted, 54 deduped | **108s** |
| Supplied markdown         | 200 — 2 extracted/deduped      | **1.5s** |

The endpoint is **functionally correct**; the issue is **execution duration / durability**. Per Cloudflare's docs, HTTP-triggered Workers have no wall-clock limit but _"when the client disconnects, all tasks associated with that request are canceled."_ So an unbounded backfill is silently canceled the moment a caller's timeout fires, and a real (`dryRun:false`) run that is cut off mid-ingest leaves a partial insert.

## Root cause (localized)

The timing data localizes the fragility to **one path**:

- The **~106s Firecrawl `scrapeOnce` in `resolveBody`** is the long pole — a single external call the worker holds the connection open for and cannot checkpoint. Extraction is cheap (~1.5s/window) and the **supplied-markdown path is already fast** (1.5s, no scrape).
- The supplied-markdown path was always the spec's _intended_ deep-history path (a local agent renders the page; the worker does cheap Haiku extraction over supplied text).
- The write path (`ingestRawReleases`) already upserts idempotently (`RELEASE_URL_UPSERT` on matching `mapEntries` slugs), so a partial insert **self-heals on re-run**. Embeds use `throwOnError:false` and missing summaries are regenerated by the autogen drain, so the write tail self-heals too.

A `Workflow` is the heavyweight fix (durability via `instanceId` + poll) but is unnecessary here: the failure is concentrated in one body-acquisition path, the deep path already works, and the write path is already idempotent. Issue #1271 acceptance criterion 2 explicitly accepts _"clearly idempotent + documented"_ as an alternative to _"durable."_

## Goals

- A default Firecrawl-auto backfill **always returns** under a normal client timeout.
- No silent truncation — when the Firecrawl ceiling bites, the report says so and tells the caller how to go deeper.
- Supplied-markdown remains the documented path for arbitrarily-deep histories (unchanged: 50 default / 200 max).
- The idempotent-re-run / self-heal property is documented so a caller disconnect is a known, recoverable state, not a surprise.

## Non-goals

- A durable `BackfillSourceWorkflow`. Recorded as a documented future escalation if real Firecrawl-auto deep histories prove necessary; not built here.
- Offset/resume on the sync endpoint (only helps the supplied-markdown path, which is already fast and caller-loopable).
- Changing the supplied-markdown / plain-fetch window budgets.

## Design

### 1. Hard ceiling on the Firecrawl path (the only code change)

In `workers/api/src/routes/workflows.ts`:

- New constant `FIRECRAWL_BACKFILL_MAX_WINDOWS = 8` (~106s scrape + ~8×1.5s extract ≈ 118s — comfortable margin under a 180s dry-run timeout).
- The clamp lives in the **route**, the only place that knows `resolved.via` — it does **not** leak into `extractChangelogAllWindows`, which stays acquisition-agnostic. After body acquisition: when `resolved.via === "firecrawl"`, the effective window budget becomes `min(requestedMaxWindows, FIRECRAWL_BACKFILL_MAX_WINDOWS)`.
- Supplied-markdown and plain-fetch paths are untouched (no scrape; caller controls depth via `maxWindows`, still clamped to the existing 1–200 range).

### 2. No silent cap

The loop already returns `cappedAtWindow` + `droppedChars`. Add an optional `guidance?: string` to `SourceBackfillReport`, populated **only** when the Firecrawl ceiling actually reduced the requested budget — e.g.:

> "Capped at 8 windows to fit the Firecrawl scrape budget. Re-run with `markdown` supplied (render the page yourself) to backfill deeper history."

Additive to the wire shape — the CLI verb (releases-cli#252) is not built yet, and existing callers ignore unknown fields.

### 3. Docs

- `docs/architecture/firecrawl-monitoring.md` backfill section: state that the Firecrawl-auto path is capped at `FIRECRAWL_BACKFILL_MAX_WINDOWS` for reliable completion; supplied-markdown is the deep-history path; re-running is idempotent and repairs any partial insert or unenriched rows.
- This spec records the resolution of the predecessor spec's "durable Workflow" non-goal: the smoke test resolved to _bound + document_, with the Workflow as a future escalation.

## Error handling

Unchanged from the predecessor. The clamp is silent-safe: it can only _reduce_ work, never error. The `guidance` string and existing `cappedAtWindow`/`droppedChars` fields surface the reduction.

## Testing

Route unit tests (in-process `app.fetch`, via the existing `_backfillExtractOverride` hook):

- A `firecrawl.enabled` source requested with `maxWindows: 50` calls the extract path with an **effective ceiling of 8** (assert the clamp), and the response carries the `guidance` field.
- A **supplied-markdown** request with `maxWindows: 50` is **not** clamped (extract sees 50; no `guidance`).
- (Existing backfill route tests stay green — bare-slug 400, unknown-id 404, non-scrape 400, supplied-markdown dry-run report.)

Gates: `npx tsc --noEmit` (root + `workers/api`), `bun test`, `bun run lint`, `bun run format:check`.
