# Collection daily-summary rubric

Grades a collection's daily rollup — the `title`, one-line `summary`, and `takeaways[]`
generated by `packages/ai/src/collection-summary.ts` for one Eastern-Time day across a
curated collection's member products. The production prompt (`SYSTEM_PROMPT`) is the
canonical guide for the model; this rubric is the parallel artifact for the grader. They
overlap in spirit but exist for different readers.

The artifact is the three fields plus the day's input releases (collection name, date, and
the `org / product: title — summary` lines fed to the model). Faithfulness and consolidation
criteria are checkable against those inputs.

## Format

- Three fields present: a non-empty `title`, a non-empty one-sentence `summary`, and zero to
  five `takeaways`. More than five takeaways is a fail.
- No markup leakage in any field: no XML tags (`<title>`, `<item>`, `</…>`), no markdown
  code fences, no markdown bullets or links, no echoed input labels ("Collection:", "Date:").
- Title is 30–90 characters, sentence case, with no trailing punctuation and no surrounding
  quotation marks. **Count the title length and state it in `evidence`.**
- Summary is a single sentence (at most one interior sentence-ending mark).

## Content

- **Title characterizes the DAY, not one release.** A theme ("Labs pile on agentic coding",
  "Quiet day, one SDK bump") passes; enumerating products or counts ("Three updates from
  Anthropic") is weak. A version number anywhere in the title is a fail.
- **Summary names substance, not a count.** "Claude Code gained managed-model enforcement
  while Devin added Slack-triggered sessions" passes; "Anthropic released three updates" fails.
  The summary does not merely restate the title.
- **No hollow abstraction.** The summary names a concrete change. "The AI SDK patched SSRF and
  replay vulnerabilities" passes; "the day focused on enhancing security and adding new
  capabilities" is a fail — it conveys nothing. On a genuinely routine day, an honest
  characterization ("a maintenance day across the platform CLIs, with X the only user-facing
  change") passes; dressing churn up as meaningful news fails.
- **Title names the theme when one exists.** When two or more products ship the same KIND of
  change, the title names the trend ("Labs tighten enterprise access controls") rather than
  listing products ("OpenAI, Anthropic update admin settings"). When one item dominates, the
  title leads with it.
- **No redundant restatement across layers.** On a quiet day carried by a single release, the
  title and summary should state it and takeaways should be empty rather than saying the same
  fact a third time. Title + summary + one takeaway all restating one trivial release is a fail.
- No filler or marketing intensifiers ("game-changing", "powerful", "seamless",
  "comprehensive", "exciting"). No opening filler ("Today,", "This is").

## Consolidation (collapse releases into the change)

- **No version or release enumeration.** When a product shipped several releases that day
  (e.g. Claude Code v2.1.175 + v2.1.176 + v2.1.177), they are folded into the product's net
  change. Two bullets distinguished mainly by version number ("Product vX does A" / "Product vY
  does B") are a fail. One bullet per product is the expected default; a second bullet for the
  same product passes only when it carries a genuinely distinct, significant change.
- **SDK families consolidate.** A day where one org bumps many packages (e.g. twenty `@clerk/*`
  releases) gets ONE bullet about the real API change, not a per-package or per-version list.
- **Skip noise.** Dependency bumps, internal tooling / build-pipeline changes, checksum or
  patch-only releases, and version-only entries with no summary carry no user-facing change and
  should be omitted. A bullet whose entire content is such noise ("X received a dependency
  update") is a fail.
- **Bullets stay tight.** Aim for roughly one line each (~30 words). Consolidation means leading
  with the most important changes, not concatenating every detail with "and … and". A run-on
  bullet that chains four or more distinct changes is a fail — split or trim.
- **No vague tails.** A bullet should end on a concrete change, not trail off into "and build
  process changes" or "and various improvements". Such a tail is a fail — trim it.

## Weighting (the right things lead, by impact)

- **Products are weighted by user impact, not given equal space.** The biggest user-facing story
  of the day leads and may take one or two bullets; lesser items get less. A genuine new
  capability or security fix and a product's routine version churn cannot occupy equal bullet
  weight — that is a fail.
- **Routine churn consolidates or drops.** A product whose day was only minor releases / version
  bumps with no standout gets one terse bullet at most, or is omitted when a more important story
  needs the room. A day of ten routine CLI releases rendered as several equal bullets is a fail.
- **Honest about quiet days.** When no item across any product stands out, the artifact says so
  ("a maintenance day across the platform CLIs") rather than inflating churn into a balanced
  roundup that implies more happened than did.

## Voice (release notes, not changelog)

- **Lead with the change, not the version.** A bullet's subject is the user-facing change with
  the product as actor: "Claude Code now restricts which models are available" passes;
  "Next.js v16.3.0-canary.51 prevents a fallback flash" fails — the version is the headline
  noun. A version number may appear as a trailing aside at most, never as the subject.
- **No redundant org prefix.** When the product name already identifies it, "Claude Code" is
  correct and "Anthropic Claude Code" is redundant. The org is named only to disambiguate.
- **One factual claim per bullet**, plain language. Precise technical terms (OAuth, PTY, GC
  pressure, prepared statements) stay; buzzwords do not.
- When several products ship the same KIND of thing, one grouped bullet ("Cursor, Windsurf,
  and Copilot all added background agents") beats one bullet each.

## Faithfulness

- No confabulation. Every product, version, capability, and number in the output maps to an
  item in the day's input releases. Invented products or versions are a fail.
- No contradiction. The output never calls a deprecation an addition or a removal a feature.
- When the day's real news is thin, the artifact is short rather than padded. Omission beats
  invented substance — a two-bullet day grounded in real releases beats five bullets of filler.
