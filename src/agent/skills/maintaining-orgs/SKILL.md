# Maintaining Orgs

Keep indexed organizations current by fetching their sources and regenerating overviews. This is a local-only workflow ‚Äî run it from Claude Code with the `releases` CLI linked and remote mode active.

## When to Use

- Periodic maintenance: "update the top 10 orgs"
- Before analysis: ensure data is fresh before competitive intel or trend work
- After onboarding: first full fetch + overview for a newly added org
- Spot checks: "is Stripe up to date?"

## Single Org Update

### Step 1: Check the playbook

```bash
releases admin content playbook <slug>
```

Read the agent notes for fetch instructions, traps, and source-specific quirks. **Skip this for simple orgs** ‚Äî if the org has only `feed` and `github` sources (no `scrape` or `agent`), the playbook rarely adds actionable context. Check source types first with `releases admin org show <slug>`.

### Step 2: Fetch all sources

Run `releases admin source fetch <source-slug> --max 20` for each active source. Skip sources with `fetchPriority: paused`.

For orgs with 3+ sources, note the source count and types before fetching ‚Äî scrape sources are slower and may need `--max 10` to stay reasonable.

### Step 3: Regenerate the overview

```bash
releases admin org show <slug> --regenerate
```

This pulls releases from the last 90 days (capped at 50 by recency) and generates an AI overview. The cap means high-volume orgs may have important releases pushed out by minor GitHub tags ‚Äî see the note on release selection below.

### Step 4: Verify

Confirm the overview regenerated by checking for the "Generating overview..." log line on stderr. If no releases exist in the 90-day window, the command warns and skips generation.

## Batch Updates

When updating multiple orgs, use parallel Claude Code sub-agents (one per org). Each agent runs the single-org workflow independently.

### Selecting targets

Pick orgs by activity level. Good candidates for routine maintenance:

- **High-value orgs** with many sources (Vercel, Stripe, Cloudflare, Anthropic, GitHub)
- **Recently active orgs** ‚Äî check `recentReleaseCount` in `releases admin org list --json`
- **Orgs not recently fetched** ‚Äî sources with stale `lastFetchedAt`

Previously completed in this project: Anthropic, OpenAI, Vercel, Cloudflare, GitHub, Stripe, Supabase, Sentry, Clerk, Prisma, Datadog, Docker, Expo, Linear, Neon.

### Dispatching agents

Use the Agent tool with `run_in_background: true` and `model: "sonnet"` for each org. Send all agent calls in a single message for maximum parallelism.

Each agent prompt should include:
1. The working directory (`/Users/zachdunn/Code/released`)
2. That `releases` is bun-linked and remote mode is active
3. The org slug
4. The four-step workflow above
5. Instructions to report back concisely: sources fetched, new releases, overview status

Example prompt template:

```
You are operating the Released CLI (`releases`) ‚Äî a bun-linked changelog indexer.
Working directory: /Users/zachdunn/Code/released

Your task: fetch latest updates and regenerate the overview for **{Org Name}** (slug: `{slug}`).

Steps:
1. Check the playbook: `releases admin content playbook {slug} 2>/dev/null`
2. List sources: `releases admin org show {slug} 2>/dev/null`
3. Fetch each active source: `releases admin source fetch <source-slug> --max 20 2>/dev/null`
4. Regenerate the overview: `releases admin org show {slug} --regenerate 2>/dev/null`

Report back: which sources were fetched, how many new releases found, and whether
the overview was regenerated successfully. Keep it concise.
```

### Tracking results

After all agents complete, compile a summary table:

| Org | Sources | New Releases | Overview |
|-----|---------|-------------|----------|

This gives a quick audit of what changed and whether any orgs need attention.

## Known Limitations

### Release selection for overviews

The `--regenerate` flag takes the 50 most recent releases by date across all sources. For high-volume orgs (e.g., Vercel with many GitHub SDK repos), minor patch releases can crowd out significant product launches. This is tracked in #250. Until that's fixed, be aware that overviews for orgs with 5+ GitHub sources may underrepresent scrape/feed sources that publish less frequently but carry more signal.

### No single-command refresh (yet)

There's no `releases admin org refresh <slug>` that combines fetch + regenerate (#252). The multi-step workflow above is the current process. When #252 and #253 (`fetch --org`) ship, this skill should be updated to use them.

### Regenerate confirmation

The `--regenerate` flag doesn't print explicit confirmation (#254). Look for the "Generating overview..." log line on stderr as a proxy. If the org has no releases in the 90-day window, it logs a warning instead.
